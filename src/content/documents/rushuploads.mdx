chapters:
  - title: The upload flow waited until the sender was ready.
    claim: Senders needed a clear single-page flow where selecting files did not immediately commit the product to upload work.
    decision: Allow drag-and-drop or file picking, file removal before submission, title and message entry, expiry selection, and optional download limit. Start upload only when the sender clicks send.
    reasoning: This kept the interaction forgiving. It also let the system create pending transfers deliberately before initiating multipart upload and activating the share link.

    claim: Recipients should be able to download from a transfer page without creating an account or understanding the sender dashboard.
    decision: Make each transfer page show title, optional message, file names, file sizes, and individual download buttons. Route download clicks through a download interstitial with ads and a short countdown.
    reasoning: The recipient flow stayed simple while still creating the ad-supported moment that drove the sender earnings model. Sender downloads bypassed the interstitial, keeping ownership behavior distinct from recipient behavior.

What shipped was a temporary large-file delivery product with sender-owned transfers, public recipient pages, download interstitials, ad impressions, earnings, payout requests, admin moderation, and lifecycle cleanup.

The product sharpened a familiar product-engineering trade-off: the simplest public flow often depends on a much more careful operational system behind it.
